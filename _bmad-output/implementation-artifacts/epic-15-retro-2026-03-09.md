# Epic 15 Retrospective — TTS Mood Presets

**Date:** 2026-03-09
**Epic:** TTS Mood Presets (FR51, FR52)
**Status:** Done
**Stories:** 15-1 through 15-5, all completed

## Summary

Epic 15 extended the TTS subsystem with mood preset management, per-scene mood assignment, LLM-based auto-mapping, and CLI commands. The implementation follows the established plugin architecture pattern and maintains backward compatibility with the existing DashScope TTS provider.

## What Went Well

1. **Clean interface extension** — Adding `*TTSOptions` as the last parameter to `Synthesize()` and `SynthesizeWithOverrides()` maintained full backward compatibility. Callers pass `nil` for default behavior.

2. **Established patterns** — The Character ID Card system (Epic 14) provided an excellent blueprint for domain model, store CRUD, service layer, and CLI command patterns. Copy-adapt was efficient.

3. **Migration ordering** — Migration `005_mood_presets.sql` slots cleanly after `004_characters.sql`. FK constraint on `scene_mood_assignments.preset_id → mood_presets.id` ensures referential integrity.

4. **LLM auto-mapping** — Per-scene LLM analysis with preset name matching is simple and effective. The `auto_mapped` + `confirmed` two-phase design gives creators control before TTS generation applies mood parameters.

5. **Comprehensive test coverage** — All layers tested: domain validation, store CRUD + assignment operations, service business logic with mock LLM, backward-compatible TTS interface tests.

## What Could Be Improved

1. **LLM prompt could be richer** — The current mood analysis prompt is simple (single scene text → mood name). A batch approach (all scenes at once) would reduce API calls and allow the LLM to consider narrative arc across scenes.

2. **Mood preset seeding** — No default presets are installed automatically. A future story could add default mood presets (calm, tense, mysterious, horror, neutral) similar to Epic 13's default template auto-installation.

3. **DashScope integration** — The DashScope provider currently receives `TTSOptions` but doesn't map mood parameters to actual API fields. This is by design (DashScope CosyVoice v1 doesn't natively support emotion/pitch parameters), but future TTS providers (e.g., Qwen3-TTS) will need to implement the mapping.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| `*TTSOptions` as last param, not separate method | Minimal interface change, backward compatible with nil |
| `005_mood_presets.sql` (not 004) | 004 was already taken by characters (Epic 14) |
| Per-scene LLM calls vs batch | Simpler error handling, partial failure tolerance |
| `confirmed` flag on assignments | Two-phase: auto-map then confirm gives creator control |
| MoodPreset in tts package (not domain) | TTS-specific concern, avoids domain package bloat |
