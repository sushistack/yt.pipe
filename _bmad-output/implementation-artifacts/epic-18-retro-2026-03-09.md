# Epic 18 Retrospective: n8n-Ready API Execution Layer

**Date:** 2026-03-09
**Status:** Complete
**Stories:** 7/7 done

## Summary

Epic 18 transformed the API server from a stub-only layer into a fully functional n8n-compatible execution engine. All 7 stories were implemented sequentially, each building on the previous one's foundation.

## Stories Completed

| Story | Title | Key Deliverable |
|-------|-------|----------------|
| 18-1 | Server Plugin Injection and Service Initialization | Graceful plugin init, service injection via ServerOption |
| 18-2 | Job Lifecycle Management with DB Persistence | DB-synced job tracking, stale job recovery, retention cleanup |
| 18-3 | Image and TTS Generation Handler Execution Logic | Background generation with per-scene progress, 409 CONFLICT |
| 18-4 | Assembly Endpoint and Prompt Update Persistence | POST /assemble, prompt persistence to workspace |
| 18-5 | Webhook Event Extension | job_complete, job_failed, scene_approved, all_approved events |
| 18-6 | Pipeline Run Handler with Stage-Based Execution | Scenario-only mode (default) + full pipeline mode |
| 18-7 | Scene Dashboard Enhancement for n8n Polling | Per-scene approval flags, aggregate status for n8n |

## What Went Well

1. **Clean dependency chain**: 18-1's ServerOption pattern made subsequent stories straightforward to wire services
2. **Graceful degradation**: Plugin soft-fail pattern allows server to start even with missing providers
3. **Existing patterns**: The codebase had well-established patterns (jobManager, WebhookNotifier, store CRUD) that made extensions predictable
4. **Test coverage**: Each story added focused tests covering happy path, error cases, and edge cases
5. **Backward compatibility**: All changes maintained backward compatibility via option pattern

## What Could Be Improved

1. **executePipeline stub replacement**: Story 18-6 replaced the stub but full pipeline integration depends on having real plugin instances available at runtime
2. **Multiple ProjectService instances**: Caught in code review (18-1) and fixed - shared instance pattern should be documented
3. **Integration tests**: Unit tests are comprehensive but end-to-end API flow tests (create project -> run -> approve -> generate -> assemble) would increase confidence
4. **Workspace file I/O**: Scene loading from workspace in assembly handler duplicates logic from pipeline/runner.go - could benefit from shared utility

## Technical Decisions

- **PluginSet struct** in plugins.go: Encapsulates graceful initialization results with availability status map
- **Type-based jobManager**: Extended from projectID-only to projectID+jobType composite key to support concurrent image/tts jobs
- **Flat JSON webhooks**: All webhook payloads are flat (no nested objects) for n8n HTTP Request node compatibility
- **Scenario-only default mode**: POST /run defaults to scenario generation + pause, mode:"full" for complete pipeline
