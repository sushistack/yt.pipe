# Epic 14 Retrospective: Character ID Card System

**Date:** 2026-03-09
**Epic:** Character ID Card System (FR48-FR50)
**Stories:** 14-1 through 14-6 (6 stories, all done)

## Summary

Epic 14 implemented per-SCP character ID cards for visual consistency in image generation. Characters can be registered with canonical names, aliases, visual descriptors, style guides, and base image prompts. The system auto-matches characters in scene text during image generation to inject consistent visual references.

## What Went Well

1. **Clean interface extension**: Adding `CharacterRef` and `CharacterRefs` to the existing `GenerateOptions` struct maintained 100% backward compatibility — zero call site changes needed across 15+ locations.

2. **Established patterns**: Following the template_cmd.go, template.go, manifest.go patterns made domain model → store → service → CLI implementation straightforward and consistent.

3. **Migration numbering**: Correctly identified that 003 was taken by Epic 13 templates and used 004 instead, avoiding a migration conflict.

4. **Optional integration**: Using `SetCharacterService()` injection with nil-safe checks in ImageGenService meant zero changes to existing tests while enabling character auto-reference for new code paths.

5. **Comprehensive test coverage**: 40+ test cases across domain, store, service layers covering CRUD, validation, character matching (canonical name, alias, case-insensitive, dedup, no match).

## What Could Be Improved

1. **Architecture doc drift**: The architecture spec uses `003_characters.sql` and a separate `ImageGenOptions` type, but the actual codebase uses `003_templates.sql` and `GenerateOptions`. Docs should be updated to reflect reality.

2. **Character matching efficiency**: Current implementation loads ALL characters plus SCP-specific characters for every scene. For projects with many characters, this could be optimized with caching.

3. **SiliconFlow prompt enrichment**: Story 14-5 injects CharacterRefs into opts but the SiliconFlow provider currently ignores them. A follow-up story should compose character visual descriptors into the actual prompt string.

## Lessons for Next Epic

- Plugin interface extension via struct field addition (not signature change) is the safest approach
- Optional service injection via setter method keeps constructors backward-compatible
- Migration numbering must be verified against actual filesystem, not architecture docs
