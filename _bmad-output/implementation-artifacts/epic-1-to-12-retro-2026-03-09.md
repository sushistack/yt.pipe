## 2. What Went Well

### 2.1 Architecture Consistency
- Plugin interface pattern established in Epic 1, **perfectly reused** through Epics 8-12
- All providers (LLM, ImageGen, TTS) follow identical HTTP client / APIError / IsRetryable() pattern
- Clean dependency direction: `domain/ ← all packages`, zero circular imports
- CLI-API dual interface shares identical service layer — zero code duplication

### 2.2 Incremental Build Strategy
- SHA-256 hash-based skip detection (Epic 5) → selective scene regeneration (Epic 9-10) → checkpoint-based stage skip (Epic 12)
- Critical for API cost optimization — only changed scenes regenerated
- Scene dependency chain (Epic 5.4) with targeted cascade invalidation

### 2.3 Defensive Programming
- Atomic file writes (temp file + rename) across all workspace operations
- Checkpoint/resume at every pipeline stage with recovery command generation
- Signal handling with checkpoint-before-exit (Epic 5.6)
- Retry with exponential backoff + jitter, error classification (retryable vs non-retryable)

### 2.4 Testing Strategy
- Mock-first approach: all external APIs mocked, real integration deferred to Epic 12.4
- Tests complete in seconds without network I/O
- Testify/mock library used consistently
- Table-driven tests, `t.TempDir()`, `t.Setenv()`, `httptest.NewServer` — proper isolation

### 2.5 Code Review Culture
- 16 review findings across Epics 1-4, **all resolved immediately** (zero carry-forward)
- 7 HIGH issues fixed: missing validation, type errors, state handling
- Zero escalations or showstoppers documented across all 60 stories

### 2.6 Multi-Stage Pipeline Patterns
- Scenario: 4-Stage (Research→Structure→Writing→Review) — most mature quality
- Image Prompts: 3-Stage (Scene→Shot→Prompt) with Frozen Descriptor Protocol
- TTS Pronunciation: 2-Tier (Glossary deterministic → LLM context-aware)
- Each stage output checkpointed as intermediate artifact for resume capability

## 8. Cross-Epic Pattern Insights

### Patterns That Worked
- **Factory + Registry pattern** for plugins — enabled runtime provider selection
- **Atomic file writes** (temp + rename) — zero data corruption incidents
- **Context propagation** — all async operations respect cancellation
- **Mock-first testing** — fast test suite, independent of external services
- **Exponential backoff with jitter** — no thundering herd issues with external APIs

### Patterns to Evolve
- **Unidirectional pipelines** → Need feedback loops for quality improvement
- **Manual quality gates** → Need automated quality evaluation
- **Silent state transitions** → Need user-visible progress and guidance
- **Word-count text processing** → Need semantic-aware Korean text handling

---

## 9. Lessons Learned

1. **Multi-stage prompt chaining produces the best quality** — The 4-stage scenario pipeline consistently outperformed single-pass approaches. Apply this pattern universally.
2. **Plugin interfaces should be defined early** — Epic 1.3's interface definitions enabled parallel development across Epics 8-12 with zero interface changes.
3. **Incremental builds are essential, not optional** — Hash-based skip detection and checkpoint/resume prevented expensive API re-calls throughout the project.
4. **Code review findings should never carry forward** — The practice of immediate resolution prevented debt accumulation.
5. **User experience needs explicit design attention** — Technical completion ≠ user-friendly. Process visibility and guidance require deliberate effort.
6. **Korean language processing needs specialized solutions** — Generic text processing approaches (regex, word-count) fail for Korean; LLM-assisted or linguistically-aware approaches are needed.
