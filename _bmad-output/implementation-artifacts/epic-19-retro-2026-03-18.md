## Epic Summary

Epic 19 delivered two independent features that enhance the content creator workflow:

1. **EFR1 — YouTube Chapters Generation**: Auto-generates YouTube chapter timestamps from scene timing data (`yt-pipe chapters <scp-id>`)
2. **EFR2 — Glossary Auto-Expansion**: LLM-based SCP term extraction with approval workflow (`yt-pipe glossary suggest/approve <scp-id>`)

|-------|-------|--------|-----------------|

| 19-2 | Glossary Suggestion Domain Model & Storage | Done | Migration 014, domain model, CRUD store |

## What Went Well

### 1. Clean Separation of Concerns
- Story 19-1 (chapters) was completely independent from 19-2~19-4 (glossary), enabling parallel development
- Each story had clear boundaries: domain → storage → service → CLI layering

### 2. Pattern Reuse
- Glossary suggestion domain/store followed established SceneApproval pattern exactly
- LLM JSON extraction reused bgm.go/mood.go prompt construction patterns
- CLI commands followed existing assemble_cmd.go and mood_cmd.go patterns

### 3. No Breaking Changes
- All features are additive — no existing pipeline behavior was modified
- `image_validation.enabled=false` pattern preserved (features are opt-in via new commands)

### 4. Pre-existing Issues Fixed
- Added `CompleteWithVision` to MockLLM and mockLLMForBGM (from Epic 20 LLM interface change)
- Updated schema version test to account for migration 015 from Epic 20

## What Could Be Improved

### 1. Manifest Missing Mood/VisualDesc
- Scene manifest.json doesn't include Mood or VisualDescription fields
- Story 19-1 had to load from scenario.json instead of manifest.json for chapter titles
- **Action Item**: Consider adding Mood/VisualDesc to scene manifest for future features

### 2. Glossary AddEntry() Was Missing
- Constraint specified using `Glossary.AddEntry()` but the method didn't exist
- Had to add it during Story 19-4 implementation
- **Action Item**: Ensure PRD constraints reference actually existing APIs, or explicitly note new methods to create

### 3. Schema Version Coordination
- Story 19-2 created migration 014, but Epic 20 had already created 015 in a parallel worktree
- Required adjusting expected schema version from 14 to 15
- **Action Item**: Multi-epic parallel work should coordinate migration numbering

## Technical Decisions

| Decision | Rationale |
|----------|-----------|
| Use `SceneScript` (from scenario.json) for chapter titles | manifest.json lacks Mood/VisualDesc; scenario.json has full scene metadata |
| Inline LLM prompt (no template file) | Follows mood.go/bgm.go pattern; prompt is short and self-contained |
| `--all` / `--select` flags for approve | Non-interactive CLI pattern; avoids stdin prompts for automation |
| `truncateRunes` helper (not reusing `truncate`) | Name collision with fact_coverage.go; rune-aware for Korean text |
| Terminal states for suggestions | `approved` and `rejected` are final — no re-approval flow needed |

## Lessons for Future Epics

1. **Independent features first**: When an epic has unrelated stories (like chapters vs glossary), implement the independent one first to get quick wins
2. **Check for name collisions**: Before adding unexported helpers, grep the package for existing names
3. **Parallel epic coordination**: Migration numbers and mock updates need coordination when multiple epics modify shared infrastructure
4. **Scenario.json is the rich data source**: For any feature needing scene metadata beyond basic paths/durations, use scenario.json not manifest.json
