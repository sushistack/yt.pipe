# Epic 17 Retrospective — BGM Preset Library

**Date:** 2026-03-09
**Epic:** BGM Preset Library (FR57-FR60)
**Status:** Done
**Stories:** 17-1 through 17-6, all completed

## Summary

Epic 17 implemented a full BGM (Background Music) preset library system: mood-tagged BGM storage, LLM-based auto-recommendation per scene, CapCut timeline integration with volume/fade/ducking controls, license metadata tracking with auto-credits, and CLI management commands. The implementation follows the established plugin architecture pattern from Epics 14 and 15.

## What Went Well

1. **Mature pattern reuse** — By Epic 17, the extension pattern (domain model → migration → store CRUD → service → assembler integration → CLI) is well-established. Stories 14 (Character ID Card) and 15 (Mood Presets) provided direct blueprints. Copy-adapt was highly efficient across all 6 stories.

2. **Clean interface extension (Story 17-1)** — Extending `AssembleInput` with `BGMAssignments []BGMAssignment` and `Credits []CreditEntry` fields maintained full backward compatibility. Zero call-site changes, zero test modifications. The Epic 14 pattern (extend struct, not signature) proved its value again.

3. **Architecture-to-code adaptation** — The architecture doc specified a different Assembler signature (`Assemble(ctx, *Project, *AssembleOptions)`), but the actual codebase uses `AssembleInput` struct. The dev correctly identified this discrepancy and followed the existing code pattern rather than the outdated doc, preventing breaking changes.

4. **Migration ordering discipline** — Migration `006_bgms.sql` slotted cleanly after `005_mood_presets.sql`. The sequential migration numbering system with embedded SQL (`//go:embed migrations/*.sql`) continues to work reliably.

5. **Comprehensive store layer (Story 17-3)** — `SearchByMoodTags` with SQL CASE/SUM ranking for multi-tag relevance is a clean solution. The upsert pattern for `AssignBGMToScene` using `ON CONFLICT DO UPDATE` handles reassignment elegantly. Delete-with-assignment protection prevents orphaned references.

6. **BGM track in CapCut format (Story 17-5)** — Separate audio track for BGM with dB-to-linear volume conversion (`dbToLinear`), per-scene timeline alignment, and proper material/segment structure. The `buildBGMTrack` function is self-contained and doesn't pollute the main track building logic.

7. **Credits integration** — `AppendBGMCredits()` in `assembler.go` merges BGM credits with existing CC-BY-SA copyright notices. Deduplication via `seen` map in `GetCredits()` prevents duplicate entries when the same BGM is used across multiple scenes.

8. **Test coverage** — All layers tested: domain validation (3 tests), store CRUD + tag search + assignments (14 tests), service with mock LLM (7 tests), CLI build verification. All 20 packages pass.

## What Could Be Improved

1. **`boolToInt` duplicate** — Story 17-3 initially introduced a duplicate `boolToInt` function that already existed in `template.go` within the same package. This was caught during compilation but could have been avoided by checking for existing utility functions first. This is a recurring pattern when multiple domain store files share helper functions.

2. **SearchByMoodTags SQL injection risk** — The current implementation uses `fmt.Sprintf` to embed tag values directly into SQL LIKE patterns (`'%%"%s"%%'`). While mood tags are internally generated (not direct user input), this could be safer with parameterized queries. Consider refactoring if tag values ever come from untrusted sources.

3. **Story file completeness** — Only stories 17-1 and 17-2 have dedicated story files in the implementation artifacts. Stories 17-3 through 17-6 were implemented directly without creating story specification files first. This reduces traceability for future retrospectives.

4. **LLM auto-recommendation prompt** — The `AutoRecommendBGMs` prompt is functional but basic. A batch approach (all scenes at once with narrative arc consideration) would produce better musical flow. The per-scene approach doesn't consider transitions between scenes.

5. **No default BGM presets** — Unlike Epic 13 (default templates), there's no auto-seeding of example BGMs during setup. Users start with an empty library and must manually add BGMs before auto-recommendation works.

6. **Architecture doc drift** — The architecture document's OutputAssembler section still describes the old signature pattern. This should be updated to reflect the actual `AssembleInput` struct pattern used since Epic 11.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Extend `AssembleInput` struct, not method signature | Follows Epic 14 pattern; zero breaking changes |
| `006_bgms.sql` (not 005) | 005 already taken by mood_presets (Epic 15) |
| SQL LIKE for JSON tag search | SQLite lacks native JSON array functions; LIKE with `%%"tag"%%` is simple and effective |
| CASE/SUM for tag ranking | Enables multi-tag relevance ordering without complex SQL |
| Separate BGM audio track in CapCut | Keeps BGM separate from narration for independent editing |
| dB-to-linear conversion in types.go | CapCut uses linear volume (0.0-1.0), BGM params stored as dB |
| `WithBGMService()` optional setter | BGM integration is opt-in; assembler works without BGM service |
| Per-scene LLM calls (not batch) | Simpler error handling, partial failure tolerance |
