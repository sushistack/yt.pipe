# Epic 16 Retrospective: Scene Approval Workflow

**Date:** 2026-03-09
**Status:** Complete
**Stories:** 5/5 done

## Summary

Epic 16 implemented per-scene image and TTS approval workflow with state machine extension, enabling creators to review and approve/reject generated assets at scene granularity before final assembly.

## What Was Delivered

### Story 16-1: State Machine Extension & Domain Model
- Extended project state machine: `approved → image_review → tts_review → assembling`
- Maintained backward compatibility via `StatusGeneratingAssets` (deprecated)
- Created `SceneApproval` domain model with status flow: `pending → generated → approved/rejected`
- Migration `007_scene_approvals.sql` with composite PK and CHECK constraints

### Story 16-2: Scene Approval Store
- Full CRUD operations: Init, MarkGenerated, Approve, Reject, Get, ListByProject
- `AllApproved()` gate function for state transitions
- `BulkApproveAll()` for --skip-approval mode
- 100% test coverage

### Story 16-3: Approval Service
- Business logic layer with status transition validation
- `AutoApproveAll()` for skip-approval mode with logging
- `GetApprovalStatus()` for dashboard summaries
- Full validation of approval state transitions

### Story 16-4: Pipeline Orchestrator Integration (Highest Risk)
- Added `SkipApproval` to `RunOptions`
- `--auto-approve` implies `--skip-approval` for backward compatibility
- Skip-approval path: legacy parallel generation (no behavior change)
- Approval path: sequential image → pause → TTS → pause → assemble
- `Resume()` handles `approved`, `image_review`, `tts_review` states
- All existing tests pass unchanged

### Story 16-5: Scene Asset Mapping Dashboard
- CLI: `yt-pipe scenes <project-id>` with table and detail views
- CLI: `yt-pipe scenes approve/reject` for per-scene approval
- API: `GET /api/v1/projects/{id}/scenes` for JSON dashboard
- API: `POST /api/v1/projects/{id}/scenes/{num}/approve|reject`

## What Went Well

1. **Backward Compatibility Preserved** — The `StatusGeneratingAssets` path is fully preserved. All existing tests pass without modification via the `--auto-approve` flag implying `--skip-approval`.

2. **Clean Layered Architecture** — Domain model → Store → Service → Pipeline → CLI/API follows the established codebase patterns perfectly.

3. **Risk Mitigation** — The highest-risk concern (state machine change + pipeline integration) was handled by keeping the old path alongside the new one rather than replacing it.

4. **Migration Ordering** — Correctly used `007_scene_approvals.sql` (after `006_bgms.sql`), avoiding conflicts.

## What Could Be Improved

1. **Approval Flow UX** — The current flow requires manual `yt-pipe run --resume` after approving all scenes. A watcher/auto-resume mechanism would improve UX.

2. **Scene Dashboard Text Excerpts** — Text excerpts from scenario are not populated in the dashboard because the scenario is stored as a file, not in the DB. Future work could load excerpts from workspace files.

3. **Rejection with Regeneration** — The current implementation tracks rejection status but doesn't auto-trigger regeneration. The creator must manually run `yt-pipe image generate` or `yt-pipe tts generate` for rejected scenes.

## Technical Debt

- `StatusGeneratingAssets` is deprecated but still referenced in several places. Once all existing projects migrate, the old path can be removed.
- The `assembler_test.go` helper `transitionToGeneratingAssets` uses the legacy path; could be updated to use the new approval flow.
