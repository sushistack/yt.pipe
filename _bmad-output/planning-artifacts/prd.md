# Product Requirements Document - youtube.pipeline

**Author:** Jay
**Date:** 2026-03-07

## Executive Summary

youtube.pipeline은 1인 SCP Foundation 유튜브 크리에이터를 위한 도메인 특화 콘텐츠 제작 파이프라인이다. SCP ID 하나를 입력하면, 최종 검수 단계의 영상 에셋을 자동으로 생성한다.

시나리오 생성, 이미지 프롬프팅 및 생성, TTS 나레이션, 자막 생성, CapCut 프로젝트 조립까지 전 과정을 워크플로우 오케스트레이션 기반의 모듈식 플러그인 아키텍처 위에서 자동 실행한다. 기술 부채가 누적된 기존 video.pipeline의 교훈을 반영해 처음부터 체계적으로 재설계한 시스템이다. MVP는 핵심 일직선 파이프라인(SCP ID → CapCut 프로젝트)에 집중하며, 플러그인 확장성과 고급 기능은 검증된 코어 위에 점진적으로 구축한다.

핵심 철학은 **"80% 자동화, 20% 수동 마무리"** — 파이프라인이 모든 생성 및 조립 작업을 처리하고, 크리에이터는 타이밍 조정, 컷 전환/효과 추가, 최종 검수에만 집중한다. 422개 크롤링된 SCP 구조화 데이터(facts.json, meta.json, main.txt)를 기반으로 팩트 검증, SCP 전문 용어 사전 기반 자막 정확도, 범용 도구로는 불가능한 품질 게이트를 내장한다.

영상당 제작 시간을 기존 대비 75% 이상 단축하는 것을 목표로 한다. 증분 빌드로 변경된 씬만 재생성하고, 출력 프로젝트에 가이드 마커를 삽입하여 검수 필요 지점을 자동 표시한다. 이미지 생성의 품질 일관성은 비주얼 ID카드와 스타일 프리셋, 저해상도 스토리보드 프리뷰 후 승인 프로세스로 관리한다. 트렌드 기반 SCP 추천으로 경쟁적인 SCP 유튜브 시장에서 속도 우위를 확보한다.

### 차별화 포인트

**핵심 차별화:**
- **SCP 도메인 특화** — 422개 초기 구조화 데이터 기반(7,000+ SCP 항목으로 확장 가능), facts.json 대비 팩트 검증 내장, 도메인 특화 용어 사전으로 정확한 자막·나레이션
- **End-to-End 자동화** — SCP ID 입력 → 시나리오 → 이미지 → TTS → 자막 → CapCut 프로젝트, 최소한의 수동 개입
- **증분 빌드 시스템** — 씬 단위 재생성 및 부분 재실행으로 전체 파이프라인 재실행 불필요

**부가 차별화:**
- **가이드 마커** — 출력 프로젝트에 검수 필요 지점 자동 주석
- **트렌드 대응 제작** — 수요 기반 SCP 추천 엔진으로 트렌딩 주제 빠른 콘텐츠 생산
- **플러그인 아키텍처** — TTS 엔진, 이미지 생성기, LLM 제공자, 출력 포맷(CapCut 등)을 설정으로 교체 가능 — 외부 도구 의존 리스크 격리
- **점진적 에셋 축적** — 검증된 프롬프트, 이미지, 설정을 에피소드마다 라이브러리에 축적하여 제작 품질과 속도가 사용할수록 향상되는 성장형 시스템

## 성공 기준

### 사용자 성공

- **제작 시간 75% 단축** — 영상 1편당 8시간 → 2시간 이하 (파이프라인 자동 실행 + 수동 마무리 포함)
- **"CapCut 열었더니 거의 다 돼 있다" 경험** — 파이프라인 출력물을 열었을 때 타이밍 조정과 효과 추가만으로 완성 가능
- **빠른 회전률 달성** — 주 2~3편 제작 가능한 생산 속도
- **이미지 품질보다 속도 우선** — "적당히 괜찮으면" 바로 진행, 마음에 안 드는 씬만 부분 재생성

### 비즈니스 성공

- **콘텐츠 생산량** — 주 2~3편, 월 8~12편 안정적 생산
- **채널 성장** — 3개월 내 구독자 1만 달성
- **API 비용 효율** — 시나리오 생성은 프론티어 LLM(최고 품질), 이미지 생성은 SiliconFlow(비용 효율), 플러그인 구조로 필요시 교체 가능
- **지속 가능한 제작 루틴** — 번아웃 없이 꾸준히 돌릴 수 있는 워크로드

### 기술 성공

- **파이프라인 안정성 99.9%** — 한 번 실행하면 에러 없이 완료
- **증분 빌드 동작** — 수정한 씬만 재생성, 전체 재실행 불필요
- **SCP 데이터 즉시 추가** — 기존 크롤링 도구로 수집 → 파이프라인에 바로 투입
- **모델 교체 용이성** — 이미지 생성기(SiliconFlow → 다른 모델) 설정 변경만으로 교체 가능

### 측정 가능한 결과

| 지표 | 현재 | 3개월 목표 |
|------|------|-----------|
| 영상당 제작 시간 | 8시간 | 2시간 이하 |
| 주간 생산량 | 불규칙 | 2~3편 |
| 구독자 수 | - | 1만 |
| 파이프라인 성공률 | - | 99.9% |
| 수동 개입 비율 | 100% | 20% 이하 |

## 사용자 저니

### 저니 1: 크리에이터 — 영상 제작 성공 경로

**인물:** Jay, SCP Foundation 유튜브 크리에이터. 주 2~3편 영상 제작을 목표로 하는 1인 콘텐츠 제작자.

**Opening Scene:**
화요일 저녁, Jay는 이번 주에 올릴 SCP 영상을 만들려고 한다. 예전엔 시나리오 쓰는 데만 3시간, 이미지 찾고 만드는 데 2시간, 편집에 3시간 — 총 8시간이 걸렸다. 오늘은 youtube.pipeline을 연다.

**Rising Action:**
1. Jay는 CLI에서 SCP-173을 입력한다. 파이프라인이 facts.json, meta.json, main.txt를 자동으로 로딩한다.
2. 프론티어 LLM이 SCP-173의 구조화된 데이터를 기반으로 시나리오를 생성한다 — 인트로, 격리 절차, 설명, 사건 기록, 결론 구조로.
3. **시나리오 리뷰 단계** — 생성된 시나리오가 마크다운 파일로 열린다. Jay가 내용을 확인하고, "격리 절차 부분을 좀 더 긴장감 있게 해줘"라고 수정 지시를 내리면 LLM이 해당 섹션만 재생성한다. 만족하면 승인.
4. **이미지 생성 + 씬별 승인** — 승인된 시나리오 기반으로 이미지 프롬프트가 자동 생성된다. 캐릭터 ID카드가 있는 개체는 자동 참조된다. 1씬씩 이미지를 생성하고, Jay가 미리보기로 확인 후 승인/재생성을 결정한다. API 과금 제어와 품질 확인을 동시에 수행한다.
5. **TTS 합성 + 씬별 승인** — 씬별 분위기 프리셋(공포, 긴장 등)을 지정한 후, 1씬씩 나레이션을 합성한다. Jay가 각 씬의 나레이션을 미리듣기로 확인 후 승인/재합성을 결정한다. 자막이 나레이션 기반으로 자동 생성된다.
6. **BGM 매핑** — 시나리오 분위기에 맞는 BGM을 프리셋 라이브러리에서 자동 추천하고, Jay가 확인·조정한다.
7. 모든 에셋(이미지, 나레이션, 자막, BGM)이 CapCut 프로젝트로 자동 조립된다.

**Climax:**
Jay가 CapCut을 연다. 나레이션 위에 이미지가 배치되고, 자막이 동기화되어 있다. "와, 진짜 거의 다 돼 있네." 타이밍 몇 군데 조정하고, 씬 전환에 효과 추가하고, 최종 검수만 하면 된다.

**Resolution:**
2시간 만에 영상이 완성됐다. 예전에는 하루 종일 걸렸는데, 이제 저녁 시간에 한 편을 뚝딱 만들 수 있다. 이 속도면 주 3편도 가능하다.

**이 저니가 드러내는 요구사항:**
- SCP 데이터 자동 로딩 시스템
- LLM 기반 시나리오 생성 + 마크다운 기반 리뷰/수정 인터랙션
- 프롬프트 템플릿 저장소 (시나리오/이미지/TTS 카테고리별 관리)
- 캐릭터 ID카드 기반 이미지 프롬프트 자동 참조
- 이미지 프롬프트 자동 생성 + SiliconFlow 연동
- 이미지/TTS 씬별 생성-확인-승인 워크플로우 (API 과금 제어)
- TTS 분위기 프리셋 설정 + 합성 + 자막 생성
- BGM 프리셋 라이브러리 + 자동 매핑
- CapCut 프로젝트 자동 조립 (BGM 포함)

### 저니 2: 크리에이터 — 에러 및 수정 경로

**인물:** Jay, 파이프라인 실행 중 결과물이 마음에 들지 않거나 에러가 발생한 상황.

**Opening Scene:**
SCP-682 영상을 만드는 중이다. 파이프라인이 끝까지 돌았는데, 결과를 보니 문제가 있다.

**Rising Action — 시나리오 수정:**
시나리오의 "파괴 불가" 설명이 너무 단조롭다. Jay는 시나리오 리뷰 단계로 돌아가 "파괴 시도 에피소드를 3개 구체적으로 넣어줘"라고 지시한다. LLM이 해당 섹션만 재생성한다. 나머지 시나리오는 그대로.

**Rising Action — 이미지 불만족:**
씬-이미지 매핑 목록을 확인한다: "씬 5: 격리 절차 — [이미지 미리보기]". 5번째 씬의 이미지가 SCP-682 외형과 맞지 않는다. Jay는 해당 씬의 이미지 프롬프트를 수정하고 — 캐릭터 ID카드의 SCP-682 외형 묘사가 자동 참조된다 — 5번 씬만 이미지를 재생성한다. 나머지 9개 씬은 건드리지 않는다. 각 씬의 산출물은 `project/{scp-id}/scenes/{scene-num}/` 하위에 독립 저장되어 있어 부분 재생성이 가능하다. 1씬 단위 승인 워크플로우 덕분에 불필요한 API 과금 없이 문제 씬만 수정한다.

**Rising Action — TTS 발음 문제:**
"변칙 개체"라는 단어의 발음이 이상하다. Jay는 TTS 설정에서 해당 단어의 발음을 교정하고, 나레이션을 해당 구간만 재합성한다.

**Climax:**
수정이 필요한 부분만 부분 재실행하니, 전체를 다시 돌리지 않아도 된다. 20분 만에 수정 완료.

**Resolution:**
에러 복구에 들인 시간이 전체 제작 시간의 일부에 불과하다. 예전처럼 처음부터 다시 시작하는 일은 없다.

**이 저니가 드러내는 요구사항:**
- 시나리오 섹션별 부분 재생성
- 씬-이미지 매핑 목록 (어떤 이미지가 어떤 씬인지 즉시 파악)
- 씬 단위 이미지 재생성 (증분 빌드)
- 씬별 독립 저장 구조 (`project/{scp-id}/scenes/{scene-num}/`)
- TTS 발음 교정 + 구간 재합성
- 파이프라인 중간 산출물 보존 (전체 재실행 방지)
- 단계별 에러 로그 및 상태 표시

### 저니 3: 시스템 관리자 — 설정 및 플러그인 관리

**인물:** Jay, 시스템의 AI 모델이나 글로벌 설정을 변경하려는 상황.

**Opening Scene:**
SiliconFlow의 이미지 품질이 아쉬워서 다른 이미지 생성 모델로 바꿔보고 싶다. 또한 새로 나온 TTS 모델이 한국어 발음이 더 자연스럽다는 소식을 들었다.

**Rising Action:**
1. Jay는 글로벌 설정 파일을 연다.
2. 이미지 생성 플러그인을 SiliconFlow에서 다른 제공자로 변경한다 — 엔드포인트 URL, API 키, 모델명만 바꾸면 된다.
3. TTS 플러그인도 마찬가지로 설정 변경만으로 교체한다.
4. 테스트 SCP 하나로 파이프라인을 돌려 새 모델의 출력을 확인한다.

**Climax:**
설정 파일 수정 → 테스트 실행 → 결과 확인. 코드 수정 없이 모델 교체 완료.

**Resolution:**
더 나은 모델이 나올 때마다 설정만 바꿔서 바로 적용할 수 있다. 파이프라인은 모델에 종속되지 않는다.

**이 저니가 드러내는 요구사항:**
- 플러그인 설정 파일 구조 (엔드포인트, API 키, 모델명)
- 글로벌 설정 프로필 시스템
- 프로젝트별 설정 오버라이드
- 설정 변경 후 테스트 실행 기능

### 저니 4: n8n 워크플로우 — API 소비자

**인물:** n8n 오케스트레이터, 파이프라인의 각 단계를 API로 호출하는 자동화 시스템.

**Opening Scene:**
Jay가 설정한 n8n 워크플로우가 트리거된다. SCP ID가 입력으로 들어온다.

**Rising Action:**
1. n8n이 데이터 로딩 API를 호출 → SCP 데이터 반환
2. 시나리오 생성 API 호출 → 시나리오 JSON 반환
3. (시나리오 승인 대기 — 프로젝트 상태가 `scenario_review`로 전환, Jay에게 알림 → 승인/수정)
4. 이미지 생성 API 호출 → 씬별 이미지 경로 반환
5. TTS API 호출 → 오디오 파일 경로 반환
6. 자막 생성 API 호출 → 자막 파일 경로 반환
7. CapCut 조립 API 호출 → 프로젝트 파일 경로 반환
8. 프로젝트 상태가 `complete`로 전환, 완료 알림을 Jay에게 전송

**Climax:**
각 API가 명확한 입출력 규약을 따르므로, n8n에서 단계별 상태 모니터링, 에러 시 자동 재시도, 병렬 처리 등을 자유롭게 구성할 수 있다.

**Resolution:**
파이프라인의 오케스트레이션이 n8n에 위임되어, 배치 처리, 스케줄링, 에러 핸들링 등을 파이프라인 코드 변경 없이 n8n 워크플로우에서 관리한다.

**이 저니가 드러내는 요구사항:**
- 각 단계별 독립 API 엔드포인트
- 명확한 입출력 JSON 규약
- 단계별 상태 코드 및 에러 응답
- 프로젝트 상태 머신 (`pending → scenario_review → approved → generating → complete`)
- 승인 대기 상태 지원 (비동기 워크플로우)
- 진행률 조회 API

### 저니 5: 크리에이터 — 온보딩/첫 사용

설치부터 첫 실행까지 10분. 결과물의 품질은 완벽하지 않아도, 전체 파이프라인이 작동한다는 확신을 얻었다. 이제 설정을 자기 취향에 맞게 조정하면서 본격적으로 사용할 준비가 됐다.

**이 저니가 드러내는 요구사항:**
- 초기 설정 위저드 (API 키, 데이터 경로, 기본 프로필)

- 기본값 프리셋 (첫 실행 시 최소 설정으로 동작)
- 테스트 실행 기능

## 도메인 특화 요구사항

### 라이선스 및 저작권

- **CC-BY-SA 3.0 자동 포함** — SCP Foundation 콘텐츠는 CC-BY-SA 3.0 라이선스 하에 배포됨. 파이프라인이 영상 설명(description)에 저작권 표기를 자동 삽입
- **AI 생성 콘텐츠 라벨링** — AI 생성 이미지/나레이션 표기는 수동 관리 (플랫폼 정책 변화에 유연 대응)
- **SCP 저작권 플래그** — 특정 SCP의 추가 저작권 조건(별도 라이선스, 사용 제한)이 있는 경우 파이프라인이 경고 표시

### 기술적 제약

- **API 의존성 격리** — LLM, TTS, 이미지 생성 모두 외부 API 의존. 플러그인 아키텍처로 제공자 장애 시 대체 제공자로 전환 가능
- **데이터 무결성** — facts.json, meta.json의 구조가 파이프라인 기대 스키마와 일치해야 함. 스키마 버전 관리로 호환성 검증
- **데이터 스키마 버저닝** — facts.json/meta.json에 스키마 버전 필드 포함. 파이프라인 실행 시 호환성 체크하여 불일치 시 명확한 에러 메시지 출력
- **프로젝트 격리** — 각 SCP 프로젝트는 `project/{scp-id}-{timestamp}/` 구조로 독립 관리. 프로젝트 간 데이터 오염 방지

### SCP 도메인 특성

- **용어 사전 기반 정확도** — SCP 전문 용어(격리 등급, 특수 용어, 재단 조직명 등) 사전을 유지하여 TTS 발음 교정 및 자막 정확도 보장
- **인라인 팩트 태깅** — 시나리오 생성 시 `[FACT:key]내용[/FACT]` 형태로 facts.json 출처 태깅. 팩트 검증 시스템이 태그 기반으로 정확도 검증
- **팩트 커버리지 임계값** — facts.json 핵심 항목 대비 시나리오 내 팩트 커버리지 80% 이상 달성 시 통과. 미달 시 경고 및 보충 제안

### 프롬프트 템플릿 관리

- **프롬프트 템플릿 저장소** — 파이프라인의 모든 프롬프트(시나리오/이미지/TTS)를 카테고리별로 분류·버전 관리하는 중앙 템플릿 저장소. 프롬프트 CRUD, 이력 추적, 롤백, 프로젝트별 오버라이드 지원
- **카테고리 구분:**
  - `scenario/`: 시나리오 생성 프롬프트 (research → structure → writing → review)
  - `image/`: 이미지 프롬프트 (shot breakdown → image prompt 변환)
  - `tts/`: TTS 분위기·톤 제어 프롬프트
  - `caption/`: 시나리오 정제 (ko/en/ja) 및 번역 프롬프트
- **검증된 프롬프트 마이그레이션** — 기존 video.pipeline의 `assets/prompts/` 디렉토리에 축적된 프롬프트를 초기 템플릿 라이브러리로 마이그레이션하여 활용

### 캐릭터 일관성 관리

- **캐릭터 ID카드 시스템** — SCP 개체(캐릭터)별 외형 묘사, 스타일 가이드, 참조 이미지 프롬프트를 구조화된 ID카드로 관리. 한 번 생성·검증된 고퀄리티 캐릭터 이미지를 영속적으로 저장하고 프리셋화하여, 동일 개체가 등장하는 모든 씬에서 일관된 외형을 유지
- **Frozen Descriptor 패턴** — 기존 video.pipeline의 Entity Visual Identity + Frozen Character Descriptor 패턴을 캐릭터 ID카드 시스템의 기반으로 활용
- **캐릭터 기반 이미지 생성** — 씬별 이미지 생성 시 해당 씬에 등장하는 개체의 캐릭터 ID카드를 자동 참조하여, 캐릭터 외형 일관성을 보장하는 프롬프트를 조합

### TTS 분위기 제어

- **분위기 프리셋** — 나레이션의 톤·속도·감정·피치를 사전 정의된 프리셋(공포, 다큐멘터리, 긴장, 차분 등)으로 관리. 씬 성격에 맞는 분위기를 자동 또는 수동으로 매핑
- **Qwen3-TTS 기반 구현** — TTS 플러그인 구현 시 Qwen3-TTS 공식문서를 반드시 참조하여 분위기 제어 파라미터(감정, 속도, 피치 등)를 최대한 활용
- **VC(Voice Cloning) 옵셔널 지원** — 참조 음성을 등록하고 VC 기반 TTS 합성을 수행하는 옵셔널 경로 제공. 플러그인 인터페이스를 통해 VC 미지원 TTS 엔진에서는 기본 TTS로 폴백

### BGM 관리

- **BGM 프리셋 라이브러리** — 분위기별(공포, 긴장, 미스터리, 액션 등) BGM 파일을 등록·분류·태깅하여 관리하는 프리셋 라이브러리
- **BGM 라이선스** — 저작권 프리(로열티 프리) BGM만 사용. 각 BGM 파일에 라이선스 정보를 메타데이터로 관리하여 출력 시 자동 크레딧 포함
- **CapCut BGM 통합** — 프로젝트 조립 시 BGM 트랙을 자동 배치 (볼륨 조절, 페이드인/아웃 포함)

### 기존 에셋 활용

- **CapCut 템플릿 구조 계승** — 기존 video.pipeline의 검증된 CapCut 프로젝트 템플릿(draft.template.json, draft.meta.info.json)을 기반으로 프로젝트 생성 로직 구현. 포맷 버전 360000 (151.0.0) 고정

## CLI + API 프로젝트 타입 요구사항

### 프로젝트 타입 개요

youtube.pipeline은 **스크립터블 CLI**(주)와 **REST API 백엔드**(부)의 이중 인터페이스 구조. CLI는 직접 실행 및 n8n 등 외부 오케스트레이터에서 호출하는 자동화 진입점이며, API는 n8n 워크플로우의 HTTP 노드에서 단계별로 호출하는 오케스트레이션 인터페이스.

### CLI 구조

- **바이너리명**: `yt-pipe` (짧고 타이핑 용이, n8n 반복 호출 최적화)
- **실행 모드**: 스크립터블 (비대화형), n8n/쉘스크립트에서 호출 가능
- **출력 형식**: JSON (파싱 용이, n8n 연동 최적화)
- **설정 방식**: YAML 설정 파일 (`config.yaml`)
- **명령어 구조**:
  - `yt-pipe run <scp-id>` — 전체 파이프라인 실행
  - `yt-pipe run <scp-id> --dry-run` — API 호출 없이 파이프라인 흐름 검증
  - `yt-pipe scenario generate <scp-id>` — 시나리오 생성
  - `yt-pipe scenario approve <scp-id>` — 시나리오 승인
  - `yt-pipe image generate <scp-id> [--scene 3,5,7 | --scene 3-7]` — 이미지 생성 (전체/특정 씬, 복수 지정 지원)
  - `yt-pipe image approve <scp-id> --scene N` — 특정 씬 이미지 승인
  - `yt-pipe tts generate <scp-id> [--scene N] [--mood <preset>]` — TTS 합성 (분위기 프리셋 지정)
  - `yt-pipe tts approve <scp-id> --scene N` — 특정 씬 나레이션 승인
  - `yt-pipe character create <scp-id>` — 캐릭터 ID카드 생성
  - `yt-pipe character list` — 등록된 캐릭터 ID카드 목록
  - `yt-pipe prompt list [--category scenario|image|tts]` — 프롬프트 템플릿 목록
  - `yt-pipe prompt show <template-id>` — 프롬프트 상세 조회
  - `yt-pipe bgm list [--mood <tag>]` — BGM 프리셋 목록
  - `yt-pipe bgm assign <scp-id> [--scene N] --bgm <bgm-id>` — BGM 매핑
  - `yt-pipe assemble <scp-id>` — CapCut 프로젝트 조립
  - `yt-pipe status <scp-id>` — 프로젝트 상태 조회
  - `yt-pipe init` — 초기 설정 위저드
- **종료 코드 규약**: `0`=성공, `1`=일반 에러, `2`=설정 에러, `3`=API 에러 (n8n 분기 처리용)

### API 구조

- **인증**: API 키 기반 (헤더: `X-API-Key`)
- **데이터 형식**: JSON 요청/응답
- **응답 구조**: `{ status, data, error, timestamp, requestId }` (디버깅 추적용)
- **버저닝**: 불필요 (1인 사용, 내부 도구)
- **Rate limit**: 불필요

#### 엔드포인트 상세

| HTTP | 경로 | 설명 | 요청 바디 | 응답 |
|------|------|------|----------|------|
| POST | `/api/projects/{scp-id}/init` | 프로젝트 초기화 + 데이터 로딩 | `{ options?: {} }` | `{ projectId, state, scpData }` |
| POST | `/api/projects/{scp-id}/scenario` | 시나리오 생성 | `{ options?: { angle?, length? } }` | `{ jobId }` (비동기) |
| PUT | `/api/projects/{scp-id}/scenario/approve` | 시나리오 승인 | — | `{ state: "approved" }` |
| POST | `/api/projects/{scp-id}/images` | 이미지 생성 | `{ scenes?: [3,5,7] }` | `{ jobId }` (비동기) |
| POST | `/api/projects/{scp-id}/tts` | TTS 합성 | `{ scenes?: [N] }` | `{ jobId }` (비동기) |
| POST | `/api/projects/{scp-id}/subtitles` | 자막 생성 | — | `{ jobId }` (비동기) |
| POST | `/api/projects/{scp-id}/assemble` | CapCut 프로젝트 조립 | — | `{ jobId }` (비동기) |
| POST | `/api/projects/{scp-id}/run` | 전체 파이프라인 실행 | `{ dryRun?: false }` | `{ jobId }` (비동기) |
| GET | `/api/projects/{scp-id}/status` | 상태 조회 | — | `{ state, step, progress, elapsed }` |
| GET | `/api/jobs/{jobId}` | 작업 상태 조회 | — | `{ status, progress, result? }` |

#### 비동기 패턴

장시간 작업(시나리오/이미지/TTS/조립)은 비동기 패턴을 따른다:
1. 요청 시 즉시 `{ jobId }` 반환 (NFR3 기준 1초 이내)
2. 클라이언트는 `GET /api/jobs/{jobId}`로 폴링하여 진행률 확인
3. 작업 완료 시 `status: "completed"` + `result` 포함 응답
4. 웹훅이 설정된 경우 상태 변경 시 자동 알림 (FR30)

#### API 에러 코드

| HTTP 상태 | error.code | 설명 |
|-----------|-----------|------|
| 400 | `INVALID_REQUEST` | 잘못된 요청 파라미터 |
| 400 | `SCHEMA_MISMATCH` | SCP 데이터 스키마 불일치 |
| 401 | `AUTH_FAILED` | API 키 인증 실패 |
| 404 | `PROJECT_NOT_FOUND` | 프로젝트 미존재 |
| 404 | `SCP_NOT_FOUND` | SCP 데이터 미존재 |
| 409 | `INVALID_STATE` | 현재 상태에서 불가능한 작업 (예: 미승인 시나리오로 이미지 생성) |
| 500 | `PIPELINE_ERROR` | 파이프라인 내부 에러 |
| 502 | `API_UPSTREAM_ERROR` | 외부 API(LLM/TTS/이미지) 에러 |

### 배포 및 운영

- **Docker 패키징** — `Dockerfile` + `docker-compose.yml`로 원커맨드 배포
- **운영 환경**: 개인 홈서버 (Docker 컨테이너)
- **볼륨 마운트**: SCP 원본 데이터(`/data/raw`), 프로젝트 출력(`/data/projects`)

- **외부 의존성**: LLM API, SiliconFlow API, TTS API (네트워크 필수)

### 구현 고려사항

- **코어 로직 공유 구조**: CLI와 API가 동일한 서비스 레이어 호출
  ```
  CLI (Args Parser) → Service Layer ← API Routes
                          ↓
                    Plugin Adapters (LLM, TTS, ImageGen)
  ```
- **설정 우선순위**: CLI 플래그 > 환경변수 > 프로젝트별 YAML > 글로벌 YAML > 기본값

## 기능 요구사항

### SCP 데이터 관리

- **FR1:** 크리에이터는 SCP ID를 입력하여 해당 SCP의 구조화된 데이터(facts.json, meta.json, main.txt)를 자동 로딩할 수 있다
- **FR2:** 시스템은 로딩된 데이터의 스키마 버전을 검증하고 불일치 시 명확한 에러를 반환할 수 있다
- **FR3:** 시스템은 SCP별 프로젝트를 독립된 디렉토리 구조로 격리 관리할 수 있다

### 시나리오 생성 & 리뷰

- **FR4:** 시스템은 SCP 구조화 데이터를 기반으로 프론티어 LLM을 활용한 영상 시나리오를 자동 생성할 수 있다
- **FR5:** 시스템은 생성된 시나리오에 facts.json 출처를 인라인 태깅하여 팩트 검증을 지원할 수 있다
- **FR6:** 시스템은 시나리오의 팩트 커버리지를 설정된 임계값(기본 80%, 설정 가능) 기준으로 검증하고 미달 시 경고 및 보충을 제안할 수 있다
- **FR7:** 크리에이터는 생성된 시나리오를 마크다운 파일로 리뷰하고, 특정 섹션의 수정을 지시한 뒤, `yt-pipe scenario approve` 명령으로 승인하여 다음 단계로 진행할 수 있다
- **FR8:** 시스템은 시나리오의 특정 섹션만 재생성할 수 있다 (전체 재생성 불필요)

### 이미지 생성

- **FR9:** 시스템은 승인된 시나리오를 기반으로 씬별 이미지 프롬프트를 자동 생성할 수 있다
- **FR10:** 시스템은 설정된 이미지 생성 플러그인을 통해 씬별 이미지를 생성할 수 있다
- **FR11:** 크리에이터는 특정 씬의 이미지만 선택적으로 재생성할 수 있다 (단일/복수 씬 지정)
- **FR12:** 크리에이터는 특정 씬의 이미지 프롬프트를 수정하고 재생성할 수 있다. 해당 씬에 등장하는 개체의 캐릭터 ID카드가 존재하면 자동 참조하여 프롬프트에 반영

### TTS & 자막

- **FR13:** 시스템은 시나리오 기반으로 TTS 나레이션을 합성할 수 있다. 씬별 분위기 프리셋(속도, 감정, 피치 등)을 적용하며, Qwen3-TTS 공식문서의 분위기 제어 파라미터를 반드시 참조하여 구현
- **FR14:** 시스템은 SCP 용어 사전의 모든 항목에 대해 TTS 발음 오버라이드를 적용할 수 있다. 사전 항목 100% 적용 여부로 검증
- **FR15:** 크리에이터는 특정 구간의 나레이션만 재합성할 수 있다
- **FR16:** 시스템은 나레이션 기반으로 자막을 자동 생성할 수 있다

### CapCut 프로젝트 조립

- **FR17:** 시스템은 생성된 모든 에셋(이미지, 나레이션, 자막, BGM)을 CapCut 프로젝트로 자동 조립할 수 있다. BGM 트랙은 볼륨 조절, 페이드인/아웃을 포함
- **FR18:** 시스템은 조립된 프로젝트에 CC-BY-SA 3.0 저작권 표기를 영상 설명에 자동 포함할 수 있다
- **FR19:** 시스템은 특정 SCP의 추가 저작권 조건이 있을 경우 경고를 표시할 수 있다

### 파이프라인 제어 & 상태

- **FR20:** 크리에이터는 전체 파이프라인을 단일 명령으로 실행할 수 있다
- **FR21:** 크리에이터는 파이프라인을 단계별로 개별 실행할 수 있다
- **FR22:** 시스템은 프로젝트 상태를 상태 머신(`pending → scenario_review → approved → image_review → tts_review → assembling → complete`)으로 관리할 수 있다. `image_review`와 `tts_review` 단계에서 씬별 승인 워크플로우를 수행
- **FR23:** 크리에이터는 프로젝트의 현재 상태 및 진행률을 조회할 수 있다
- **FR24:** 시스템은 변경된 씬만 재생성하는 증분 빌드를 수행할 수 있다
- **FR25:** 시스템은 씬별 산출물을 독립적으로 저장하여 부분 재생성을 지원할 수 있다
- **FR26:** 시스템은 dry-run 모드로 실제 API 호출 없이 파이프라인 흐름을 검증할 수 있다
- **FR27:** 시스템은 각 파이프라인 단계의 실행 결과를 구조화된 로그로 기록할 수 있다
- **FR28:** 시스템은 에러 발생 시 실패 지점, 원인, 복구를 위한 CLI 명령어를 포함한 에러 정보를 제공할 수 있다
- **FR29:** 크리에이터는 씬-이미지 매핑 목록을 조회하여 각 씬의 생성된 에셋을 확인할 수 있다
- **FR30:** 시스템은 프로젝트 상태 변경 시 웹훅 알림을 전송할 수 있다. 이벤트 유형(state_changed, error, completed), 페이로드 구조(projectId, event, state, timestamp, data), 전송 실패 시 최대 3회 재시도를 지원
- **FR42:** 시스템은 파이프라인 실행 중 현재 단계명, 진행률(%), 경과 시간을 CLI에 실시간으로 표시할 수 있다
- **FR43:** 시스템은 파이프라인 실행 성공률(성공/실패 비율)을 집계하여 조회할 수 있다
- **FR44:** 시스템은 전체 파이프라인 단계 대비 수동 개입이 필요한 단계의 비율을 추적하여 조회할 수 있다

### 프롬프트 템플릿 관리

- **FR45:** 크리에이터는 프롬프트 템플릿을 카테고리별(scenario, image, tts, caption)로 등록·조회·수정·삭제할 수 있다
- **FR46:** 시스템은 프롬프트 템플릿의 변경 이력을 최근 10개 버전까지 추적하고, 특정 버전으로 롤백할 수 있다. 롤백 시 해당 템플릿을 참조하는 프로젝트에는 영향을 주지 않는다 (프로젝트는 사용 시점의 스냅샷을 보존)
- **FR47:** 크리에이터는 프로젝트별로 글로벌 프롬프트 템플릿을 오버라이드하여 커스터마이징할 수 있다

### 캐릭터 일관성

- **FR48:** 크리에이터는 SCP 개체별 캐릭터 ID카드(외형 묘사, 스타일 가이드, 참조 이미지 프롬프트)를 생성·편집할 수 있다
- **FR49:** 시스템은 캐릭터 ID카드를 영속적으로 저장하고 프리셋화하여 프로젝트 간 재사용할 수 있다
- **FR50:** 시스템은 씬별 이미지 생성 시 시나리오 텍스트에서 등록된 캐릭터 ID카드의 개체명과 매칭하여 해당 ID카드를 자동 참조하고, 일관된 외형의 프롬프트를 조합할 수 있다. 매칭은 ID카드에 등록된 개체명(정규명 + 별칭) 기반 문자열 매칭으로 수행

### TTS 분위기 & VC

- **FR51:** 크리에이터는 TTS 분위기 프리셋(공포, 다큐, 긴장, 차분 등)을 등록·관리할 수 있다. 각 프리셋은 속도, 감정, 피치 파라미터를 포함하며 Qwen3-TTS 공식문서 기반으로 구현
- **FR52:** 크리에이터는 씬별로 TTS 분위기 프리셋을 지정할 수 있다. 미지정 시 LLM 기반 시나리오 분위기 분석으로 자동 매핑하며, 자동 매핑 결과는 크리에이터에게 제시 후 확인/수정을 거쳐 확정
- **FR53:** *(Phase 2)* 시스템은 VC(Voice Cloning)를 옵셔널로 지원할 수 있다 — 참조 음성 등록, VC 기반 TTS 합성. VC 미지원 TTS 엔진에서는 기본 TTS로 폴백. 플러그인 인터페이스를 통해 구현

### 생성 승인 워크플로우

- **FR54:** 시스템은 이미지 생성 시 1씬 단위로 생성-미리보기-승인/재생성 워크플로우를 수행할 수 있다. API 과금을 제어하고 품질을 씬 단위로 확인
- **FR55:** 시스템은 TTS 합성 시 1씬 단위로 합성-미리듣기-승인/재합성 워크플로우를 수행할 수 있다
- **FR56:** 크리에이터는 씬별 시나리오 텍스트-이미지-나레이션 매핑 대시보드를 조회하여 각 씬의 전체 에셋 구성을 확인할 수 있다

### BGM 관리

- **FR57:** 크리에이터는 BGM 프리셋 라이브러리에 분위기별(공포, 긴장, 미스터리, 액션 등) BGM 파일을 등록·분류·태깅할 수 있다
- **FR58:** 시스템은 LLM 기반 시나리오 분위기 분석으로 씬/섹션별 BGM을 자동 추천할 수 있다. 추천 결과는 크리에이터에게 제시 후 확인/수정을 거쳐 확정하며, 추천 시 라이브러리 내 분위기 태그 매칭 기반으로 후보를 제시
- **FR59:** 시스템은 BGM을 CapCut 프로젝트에 자동 배치할 수 있다 (볼륨 조절, 페이드인/아웃, 나레이션 구간 볼륨 덕킹 포함). 덕킹 비율(기본 -12dB), 페이드 길이(기본 2초)는 글로벌/프로젝트별 설정으로 조정 가능
- **FR60:** 시스템은 BGM 파일 등록 시 라이선스 정보(라이선스 유형, 출처, 크레딧 텍스트)를 메타데이터로 관리하고, CapCut 프로젝트 조립 시 영상 설명에 BGM 크레딧을 자동 포함할 수 있다

### 프롬프트 마이그레이션

- **FR61:** 시스템은 초기 설정 시 기본 프롬프트 템플릿 세트(시나리오/이미지/TTS/캡션)를 자동 설치할 수 있다. 기존 video.pipeline의 검증된 프롬프트를 초기 템플릿 라이브러리로 마이그레이션하여 활용

### 설정 & 플러그인

- **FR31:** 크리에이터는 초기 설정 위저드를 통해 API 키, 데이터 경로, 기본 프로필을 설정할 수 있다
- **FR32:** 시스템은 설정된 API 키의 유효성을 검증할 수 있다
- **FR33:** 크리에이터는 YAML 설정 파일을 통해 TTS, 이미지 생성, LLM 플러그인을 교체할 수 있다
- **FR34:** 시스템은 글로벌 설정과 프로젝트별 설정 오버라이드를 지원할 수 있다
- **FR35:** 시스템은 설정 우선순위(CLI 플래그 > 환경변수 > 프로젝트별 YAML > 글로벌 YAML > 기본값)를 적용할 수 있다
- **FR36:** 크리에이터는 설정 변경 후 테스트 SCP로 파이프라인을 검증 실행할 수 있다

### API 인터페이스

- **FR37:** 시스템은 각 파이프라인 단계를 독립 API 엔드포인트로 노출할 수 있다
- **FR38:** 시스템은 API 키 기반 인증을 수행할 수 있다
- **FR39:** 시스템은 비동기 워크플로우에서 시나리오 승인 대기 상태를 지원할 수 있다. 기본 72시간 승인 타임아웃을 적용하며, 만료 시 알림을 전송한다
- **FR40:** 시스템은 일관된 JSON 응답 구조(`status, data, error, timestamp, requestId`)를 반환할 수 있다

## 비기능 요구사항

### 성능

- **NFR1:** 전체 파이프라인(SCP ID → CapCut 프로젝트) 실행 시간은 외부 API 응답 시간 제외 시 5분 이내, 외부 API 포함 전체 실행 시간은 10분 이내 (10씬 기준). 파이프라인 실행 로그의 총 경과 시간으로 측정
- **NFR2:** CLI 명령어 응답(상태 조회, 설정 검증 등 비생성 작업)은 2초 이내. 명령어 실행 시작~응답 완료 경과 시간으로 측정
- **NFR3:** API 엔드포인트 응답(요청 수신 → 작업 시작 확인)은 1초 이내. API 요청 수신~응답 전송 경과 시간으로 측정
- **NFR4:** 증분 빌드 시 변경되지 않은 씬은 재처리하지 않아 실행 시간을 (변경 씬 수 / 전체 씬 수) 비율에 비례하여 단축. 파이프라인 실행 로그의 처리 씬 수와 경과 시간으로 검증

### 안정성

- **NFR5:** 파이프라인 성공률 99.9% — 외부 API 정상 응답 조건에서 파이프라인 내부 에러로 인한 실패 없음. 최근 100회 실행의 성공/실패 비율로 측정
- **NFR6:** 외부 API 에러 시 실패한 항목만 선택적으로 자동 재시도 (최대 3회, 점진적 지연 증가)
- **NFR7:** 파이프라인 중단 시 중간 산출물 보존 — 파이프라인 단계별 체크포인트 저장, 중단 지점부터 재개 가능. 중단 후 재시작 시 이전 단계 산출물 존재 여부로 검증
- **NFR8:** 프로젝트 데이터 무결성 — 비정상 종료 시에도 기존 프로젝트 데이터 손상 방지. 비정상 종료 후 프로젝트 디렉토리의 파일 무결성(체크섬 비교)으로 검증

### 통합

- **NFR9:** 플러그인 인터페이스 표준화 — LLM, TTS, 이미지 생성 플러그인은 동일한 표준 인터페이스 규약을 준수
- **NFR10:** 외부 API 타임아웃 — 개별 API 호출 타임아웃 설정 가능 (기본 120초)
- **NFR11:** n8n 호환성 — API 응답은 n8n HTTP Request 노드에서 직접 파싱 가능한 표준 JSON 구조
- **NFR12:** CapCut 프로젝트 포맷 호환성 — 생성된 프로젝트 파일은 CapCut 포맷 버전 360000 (new_version: 151.0.0)과 호환. 기존 video.pipeline의 검증된 템플릿 JSON 구조를 기반으로 생성하며, CapCut에서 프로젝트 정상 로딩 여부로 검증

### 배포

- **NFR13:** Docker 이미지로 패키징 — `docker-compose up` 원커맨드로 전체 시스템 기동
- **NFR14:** 환경변수를 통한 API 키 주입 — 설정 파일에 시크릿 하드코딩 불필요
- **NFR15:** 데이터 영속화 — Docker 볼륨을 통해 컨테이너 재생성 시에도 SCP 데이터 및 프로젝트 출력 보존

### 보안 (최소)

- **NFR16:** API 키는 환경변수 또는 설정 파일로만 관리, 로그에 노출 금지
- **NFR17:** API 인증 실패 시 401 응답, 요청 내용 로깅하지 않음
- **NFR24:** API 서버는 기본적으로 localhost에서만 접근 가능하며, 설정을 통해 지정 네트워크로 확장 가능

### 유지보수성

- **NFR18:** 프로젝트당 디스크 사용량을 `yt-pipe status` 출력에 포함하고, 완료된 프로젝트의 중간 산출물 정리 기능 제공
- **NFR19:** 구조화된 로그는 JSON 포맷으로 출력하여 외부 로그 수집 도구와 호환 (n8n에서 파싱 가능)
- **NFR20:** 개별 모듈(LLM/TTS/이미지 생성/CapCut 조립)을 독립적으로 수정·테스트할 수 있도록 모듈 간 결합도를 최소화한다. 각 모듈의 독립 단위 테스트 실행 가능 여부로 검증
- **NFR21:** 새 플러그인 추가 시 기존 코드 변경 없이 플러그인 구현만으로 통합할 수 있다
- **NFR22:** API 상태 조회 엔드포인트는 현재 실행 중인 단계명, 진행률(%), 경과 시간을 포함하여 n8n 폴링에 최적화된 응답을 반환한다

### 테스트

- **NFR23:** 플러그인은 외부 API 호출 없이 파이프라인 전체를 단위 테스트할 수 있도록 테스트용 대체 구현을 제공한다
